You are initializing or syncing the rubot workspace for this project.

If the user provides a subfolder path as an argument, only regenerate the CLAUDE.md for that subfolder. Otherwise, run the full initialization.

**Default behavior**: CLAUDE.md is always the default output — both at root level and in subfolders. AGENTS.md is only generated when the user explicitly selects it.

## Initialization Process

### Step 1: Detect Project Information

Gather project information by examining:

```bash
# Get project name from package.json
cat package.json 2>/dev/null | grep '"name"' | head -1

# Check for project type indicators
ls -la package.json tsconfig.json wrangler.toml 2>/dev/null

# Detect framework/stack
cat package.json 2>/dev/null | grep -E "(elysia|drizzle|trpc|tanstack|react|next|vue)" | head -10

# Get git remote
git remote get-url origin 2>/dev/null
```

### Step 2: Scan Directory Structure

Map the project structure:

```bash
# List top-level directories
ls -d */ 2>/dev/null

# Find key configuration files
find . -maxdepth 2 -name "*.config.*" -o -name "*.json" 2>/dev/null | head -20

# Check for existing CLAUDE.md or AGENTS.md
ls -la CLAUDE.md AGENTS.md 2>/dev/null

# Check for existing rubot.local.yaml
ls -la .claude/rubot/rubot.local.yaml 2>/dev/null
```

### Step 3: Choose What to Generate

Use AskUserQuestion to let the user choose which files to generate (CLAUDE.md is the default):

```
AskUserQuestion({
  questions: [{
    question: "Which files would you like to generate or update?",
    header: "File Selection",
    options: [
      { label: "CLAUDE.md (default)", description: "Generate CLAUDE.md — project instructions for Claude Code. Recommended starting point." },
      { label: "Both (CLAUDE.md + AGENTS.md)", description: "Generate both files — recommended for monorepos or large projects" },
      { label: "AGENTS.md only", description: "Generate only AGENTS.md — hierarchical agent guidance for subdirectories" }
    ],
    multiSelect: false
  }]
})
```

Store the user's selection and use it to skip the irrelevant generation step (Step 7 for CLAUDE.md, Step 8 for AGENTS.md). The `rubot.local.yaml` config file is always generated/updated regardless of selection — it's the source of truth for both files.

### Step 4: Load Existing Configuration (if re-running)

If `.claude/rubot/rubot.local.yaml` already exists, read it to preserve user customizations:

```bash
cat .claude/rubot/rubot.local.yaml 2>/dev/null
```

Merge detected values with existing config — user-edited values take priority. Specifically preserve:
- `agent_browser` section (target_url, auth, scenarios)
- `project_rules` section
- `agents` section (inactive agents)
- Any custom commands

### Step 5: Create Workspace Directory

```bash
mkdir -p .claude/rubot
```

### Step 6: Generate rubot.local.yaml

Read the template and customize it for the project:

```bash
# Read template
cat ~/.claude/plugins/rubot/templates/rubot.local.yaml.template
```

Create or update `.claude/rubot/rubot.local.yaml` with:

1. **Project metadata**: name, type, repository URL
2. **Environment**: bun/node/wrangler versions detected from system
3. **Stack detection**: frameworks and libraries found in package.json
4. **Database tables**: if using Neon MCP, query actual tables
5. **Registered agents**: list of applicable agents
6. **Validation rules**: project-specific constraints
7. **agent_browser section**: target URL (default to `http://localhost:3000`), auth placeholders, and example scenarios

When updating an existing file, merge — do not overwrite user-configured values.

### Step 7: Generate CLAUDE.md

> **Skip this step** if the user selected "AGENTS.md only" in Step 3.

Create or update `CLAUDE.md` in the project root with:

Customize based on detected project structure. Read `{TARGET_URL}` from `rubot.local.yaml` `agent_browser.target_url`.

### Step 8: Generate Root AGENTS.md

> **Skip this step** unless the user explicitly selected "Both" or "AGENTS.md only" in Step 3. The default "CLAUDE.md" selection does NOT generate AGENTS.md.

Read the root template:

```bash
cat ~/.claude/plugins/rubot/templates/rootAGENTS.md.template
```

Replace placeholders using values from:
1. **rubot.local.yaml** — project name, type, git remote, commands, agent_browser config
2. **package.json** — dependencies, scripts
3. **Directory scan** — which folders exist, which have AGENTS.md files
4. **Git state** — remote URL, default branch

Key placeholders to fill from `rubot.local.yaml`:
- `{PROJECT_NAME}` → `project.name`
- `{PROJECT_TYPE}` → `project.type`
- `{GIT_REMOTE}` → `project.repository`
- `{INSTALL_COMMAND}` → `bun install`
- `{DEV_COMMAND}` → `commands.dev`
- `{BUILD_COMMAND}` → `bun run build`
- `{LINT_COMMAND}` → `commands.validate`
- `{AGENT_BROWSER_TARGET_URL}` → `agent_browser.target_url`
- `{AGENT_BROWSER_SCENARIOS}` → formatted list from `agent_browser.scenarios`

For `{AGENT_BROWSER_SCENARIOS}`, format each scenario as a numbered list:

```markdown
1. Open example.com and fill out the contact form
2. Navigate to /dashboard and verify all widgets load
3. Test the signup flow with valid and invalid inputs
```

### Step 9: Generate Sub-folder CLAUDE.md (if argument provided)

If the user provided a subfolder path, generate a **CLAUDE.md** for that subfolder (not AGENTS.md):

1. Read the sub template:
   ```bash
   cat ~/.claude/plugins/rubot/templates/subCLAUDE.md.template
   ```

2. Scan the subfolder structure:
   ```bash
   ls -la {SUBFOLDER_PATH}/
   find {SUBFOLDER_PATH} -maxdepth 2 -type f | head -30
   ```

3. Replace placeholders using:
   - `{FOLDER_NAME}` → directory name
   - `{FOLDER_PATH}` → relative path from project root
   - `{BRIEF_DESCRIPTION}` → inferred from file contents
   - `{DOMAIN}` → inferred domain (e.g., Authentication, API, UI Components)
   - `{LANGUAGE}` → detected from file extensions
   - Other placeholders from detected file structure

4. Write to `{SUBFOLDER_PATH}/CLAUDE.md`

> **Note**: If the user explicitly requests AGENTS.md for a subfolder, use `subAGENTS.md.template` instead and write to `{SUBFOLDER_PATH}/AGENTS.md`.

### Step 10: NeonDB Integration (Optional)

If the Neon MCP server is available:


Update `.claude/rubot/rubot.local.yaml` with discovered tables.

### Step 11: Verify Environment

Confirm the detected environment:

```bash
# Check tool versions
bun --version
node --version
wrangler --version 2>/dev/null || echo "wrangler not installed"

# Check if agent-browser is installed
agent-browser --version 2>/dev/null || echo "agent-browser not installed — run /rubot-setup-agent-browser"

# Check if ClawSec Suite is installed
ls -d ~/.claude/skills/clawsec-suite/ 2>/dev/null && echo "clawsec-suite: installed" || echo "clawsec-suite: not installed — run: npx skills add prompt-security/clawsec@clawsec-suite -g -y"

# Check if OpenSpec is installed
which openspec 2>/dev/null && openspec --version 2>/dev/null || echo "openspec: not installed — will be auto-installed during /rubot-plan"

# Verify validate command exists
cat package.json | grep '"validate"'
```

### Step 12: Review Agent Applicability

Use AskUserQuestion to confirm agent configuration:

```
AskUserQuestion({
  questions: [{
    question: "Which of these agents should be INACTIVE for this project?",
    header: "Agents",
    options: [

      { label: "cloudflare", description: "Skip if not deploying to Cloudflare Workers" },
      { label: "chart-master", description: "Skip if project has no data visualizations" },
      { label: "qa-tester", description: "Skip if not using agent-browser for E2E testing" },
      { label: "All agents active", description: "Keep all agents active for this project" }
    ],
    multiSelect: true
  }]
})
```

Update `.claude/rubot/rubot.local.yaml` with inactive agents.
